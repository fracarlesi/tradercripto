## 2026-01-20 - US-001: Strategy Analysis Report

### Current Strategy: TrendFollowStrategy (trend_follow.py)

**Entry Conditions:**
1. Regime must be TREND (ADX > 25)
2. ADX > min_adx (default 25)
3. Price position vs EMA200 (longs: price > EMA200, shorts: price < EMA200)
4. ATR% > 0.5% (volatility filter)
5. Direction determined by `trend_direction` from MarketState or EMA200 slope

**Exit Conditions:**
- Initial stop: 2.5 ATR from entry
- Trailing stop: 2.5 ATR from max favorable (configured in trading.yaml)
- Minimal ROI (time-based take profit) in trading.yaml

### Assets Configured

**trading.yaml - universe mode: "all"**
- Fetches ALL assets from Hyperliquid with filters
- Manual list exists (BTC, ETH, SOL) but not used since mode="all"
- Excludes stablecoins only

**intelligent_bot.yaml:**
- market_scanner with min_volume filter
- capital_allocator: max 5 positions

### Problems Identified

1. **Too Many Assets**: `mode: "all"` means trading potentially hundreds of assets instead of focusing on BTC
2. **Complex Entry Logic**: Requires TREND regime + ADX + EMA200 + ATR filter - may miss good opportunities
3. **No Simple Moving Average Strategy**: Current strategy uses EMA200 slope and breakout, not SMA crossovers
4. **No Candlestick Pattern Confirmation**: Entry relies only on indicators, no price action confirmation
5. **No Daily Trade Limit**: Nothing prevents overtrading
6. **Config Confusion**: Two YAML files with overlapping settings (trading.yaml vs intelligent_bot.yaml)
7. **Typecheck Issues**: 385 pyright errors in codebase (pre-existing technical debt)
8. **Shorts Disabled**: `allow_short: false` in trading.yaml, limiting to long-only

### Files Changed
- None (analysis only)

### Learnings for Future Iterations
- `BaseStrategy.evaluate()` returns `StrategyResult` with `has_setup` and `Setup` object
- `Setup` model requires many fields including llm_approved, llm_confidence, llm_reason
- Trading.yaml `universe.mode: "all"` is the culprit for multi-asset trading
- Strategy quality score (0-1) is calculated but meaning unclear
- Current strategy designed for "let trends run" with no take profit by default

## 2026-01-20 - US-003: Implement Simple SMA Crossover Strategy

### What Was Implemented
- Added `sma20` and `sma50` fields to `MarketState` model in `core/models.py`
- Updated `MarketStateService._fetch_market_state()` to calculate SMA20/SMA50 using `calculate_sma()`
- Completely rewrote `TrendFollowStrategy.evaluate()` to use SMA crossover logic:
  - **Golden Cross (LONG)**: Price > SMA20 AND SMA20 > SMA50
  - **Death Cross (SHORT)**: Price < SMA20 AND SMA20 < SMA50
- Simplified entry conditions - removed complex ADX/regime checks
- Strategy now works in any market regime (signals are self-contained)
- Created database migration `008_add_sma_indicators.sql` for new columns
- Updated `_save_to_db()` to persist SMA values
- Updated test fixtures with sma20/sma50 values and rewrote strategy tests

### Files Changed
- `simple_bot/core/models.py`: Added sma20, sma50 fields to MarketState
- `simple_bot/services/market_state.py`: Calculate and save SMA indicators
- `simple_bot/strategies/trend_follow.py`: Complete rewrite with SMA crossover logic
- `simple_bot/tests/test_conservative.py`: Updated fixtures and strategy tests
- `database/migrations/008_add_sma_indicators.sql`: New migration for SMA columns

### Performance Impact
- Simpler, clearer entry/exit signals
- No longer dependent on regime detection (TREND/RANGE/CHAOS)
- Shorts disabled by default (configurable via `allow_short` config)
- Quality score based on SMA separation and price proximity to SMA20

### Verification
- 193 tests pass ✓
- Pyright: 385 errors (pre-existing technical debt)

### Learnings for Future Iterations
- `calculate_sma()` returns shorter array due to `mode='valid'` - use `[-1]` for latest value

- `BaseStrategy.can_trade()` can be overridden to allow trading in any regime
- Strategy quality is now based on: SMA separation (trend strength), price proximity to SMA20 (entry timing), RSI confirmation
